# Mappningsissues och Designbeslut

Denna sida dokumenterar öppna designfrågor och fattade beslut för mappningar från svenska RIVTA-tjänstekontrakt (TKBer) till FHIR R4-resurser inom ramen för EHDS-TK. Sidan används som arbetsunderlag under implementationsguidearbetet och uppdateras löpande i takt med att frågor utreds och beslut fattas. Varje issue är kopplat till ett eller flera tjänstekontrakt och beskriver det tekniska problemet, möjliga lösningsalternativ samt aktuellt förslag eller fattat beslut. Öppna frågor kräver vidare utredning eller nationellt samordningsbeslut innan de kan avslutas.

## Öppna frågor

|---|---|---|---|---|
| ALERT-001 | GetAlertInformation | `GetAlertInformation`-bodyn innehåller ENBART `typeOfAlertInformation 1..1`. Tidigare stub-mappning antog fält som `alertCode`, `alertStatus`, `alertTimePeriod`, `causeCode`, `reaction`, `alertComment` — inget av dessa existerar i den verkliga TKBn. Designfråga: Varifrån ska detaljerade allergi/överkänslighetsfält hämtas? | Öppen | Acceptera att enbart `typeOfAlertInformation` finns tillgängligt; `AllergyIntolerance` populeras med minimalt innehåll. Kräver verifiering mot källsystemet. |
| ALERT-002 | GetAlertInformation | `typeOfAlertInformation` kodsystem är okänt/lokalt. TKBn anger "kodsystem okänt externt/lokalt". Nationellt kodverk för uppmärksamhetstyper saknas. | Öppen | Avvakta nationellt kodverk från Inera/e-Hälsomyndigheten. Tillsvidare: använd `urn:oid` för lokalt system. |
| FUNC-001 | GetFunctionalStatus | PADL-poster (`padl.typeOfAssessment` + `padl.assessment`) saknar direkt FHIR Condition-element. Tre alternativ: (1) `Observation.component`, (2) extension, (3) separata Condition-resurser. | Öppen | Koda som `Condition.note` tills vidare. Långsiktig lösning: separata Observation-resurser per PADL-post. |
| OBS-001 | GetObservations | `observationBody.observationValue.ts` är en sträng med variabel precision (`YYYYMMDD`, `YYYYMMDDHHMMSS` etc.) och kan inte mappas direkt till FHIR `dateTime`. | Öppen | Använd `valueString` eller `valueDateTime` med extension för variabelprecision. |

## Designbeslut (fattade)

| ID | TK | Beslut | Motivering |
|---|---|---|---|
| GENERAL-002 | Alla TK | OID-till-URI-mappning för `patientId.root`: fasta mappningar `1.2.752.129.2.1.3.1` → personnummer URI, `1.2.752.129.2.1.3.3` → samordningsnummer URI, `1.2.752.129.2.1.4.1` → `urn:oid` passthrough. Okänt OID behålls som `urn:oid`. | Säkerställer enhetlig och förutsägbar identifieringshantering för svenska nationella identifierare i FHIR. Återfaller på `urn:oid`-notation för att bevara information utan att förlora OID-semantiken. |
| GENERAL-003 | Alla TK | Provenance-agenter: tre agenter definieras — `custodian` (vårdgivare/yttre Sparr), `author` (vårdenhet/inre Sparr), `assembler` (EHDS-bryggan via `EHDS_BRIDGE_HSA_ID`). Gäller alla TK. | Tydlig ansvarskedja i Provenance-resursen krävs för spårbarhet och för att uppfylla PDL-krav. De tre rollerna motsvarar de juridiska aktörerna i den svenska vårdinformationsmodellen. |
| GENERAL-004 | Alla TK | Flag vs AllergyIntolerance för GetAlertInformation: Flag skapas alltid. AllergyIntolerance skapas dessutom om `typeOfAlertInformation` indikerar allergi. `Flag.extension[allergyReference]` pekar på AllergyIntolerance. | Flag täcker alla typer av uppmärksamhetsinformation inklusive icke-allergirelaterade varningar. AllergyIntolerance skapas kompletterande för allergier och överkänsligheter för att stödja FHIR-interoperabilitet med mottagarsystem som förväntar sig AllergyIntolerance-resurser. |

## Kända begränsningar

Följande begränsningar i FHIR R4 påverkar mappningarna i denna implementationsguide. Begränsningarna dokumenteras här för att ge implementatörer förståelse för var avvikelser från källmodellen är oundvikliga och för att motivera de workarounds som valts.

1. **`Observation.bodySite` är `0..1` (ej `0..*`)** — begränsar mappning av `targetSite` när flera anatomiska platser anges i källdata (se OBS-004). Workaround: extension för ytterligare platser.
2. **`DiagnosticReport` saknar inbyggt stöd för act-poster med multimedia** — fält som `actCode`, `actText` och `actResult` med multimediainnehåll från GetReferralOutcome har ingen direkt motsvarighet (se REF-001). Workaround: `DiagnosticReport.presentedForm` och extension.
3. **`MedicationStatement` saknar modellering av receptkedja (chain)** — relationen mellan insättning (`I`) och utsättning (`U`) i läkemedelshistoriken kan inte uttryckas med standardelement. Kräver extension för att bevara receptkedjans semantik (se MED-002).
4. **`DocumentReference.content[0].attachment.data` är base64** — stora dokument (t.ex. journalanteckningar med bilagor) bör refereras via URL i `attachment.url` istället för att inkluderas som inbäddad base64-data, för att undvika onödigt stora FHIR-resurser.
5. **`ImagingStudy` är komplex och kan vara overkill för enbart utlåtandemetadata** — för GetImagingOutcome där DICOM-referensdata är begränsad kan ImagingStudy-resursen bli onödigt komplex. Alternativ: inkludera DICOM-metadata direkt i `DiagnosticReport.imagingStudy`-referensen med minimalt ImagingStudy (se IMG-002).
6. **`Observation.value[x]` stödjer variabelprecisions-datum dåligt** — FHIR R4 `dateTime` kräver ett specifikt precisionsnivå, men RIVTA-tidsstämplar kan ha varierande precision (`YYYYMMDD` vs. `YYYYMMDDHHMMSS`). Ingen standardiserad FHIR-mekanism finns för att ange osäkerhetsnivå i datumvärden (se OBS-001). Workaround: `valueString` eller extension.
7. **Inga standardiserade FHIR-element för samtycke/Sparr-information** — `approvedForPatient`-flaggan i RIVTA-headrar har ingen direkt FHIR-motsvarighet. Kräver `meta.security` eller en separat Consent-resurs för korrekt semantik (se PDL-001). Nationellt beslut om tillvägagångssätt saknas.
